# 协同过滤与混合推荐：开源数据集评估与选型

本文档总结了为当前基于内容的音乐推荐系统引入**协同过滤（Collaborative Filtering）**时，可用的开源数据集选项、优缺点及落地建议。

由于本项目当前没有真实的用户-物品交互数据，引入带用户行为或歌单共现的开源数据集是实现混合推荐（Hybrid Recommendation）的最佳路径。

---

## 1. 核心目标与挑战

- **核心目标**：获取真实的群体偏好信号（听歌历史或歌单共现），训练 Item-CF 或矩阵分解模型，然后与现有的内容推荐分（CBF）进行混合排序。
- **最大挑战（ID 映射）**：当前系统库（`songs_processed.csv`）中的 `song_id` 是 **Spotify Track ID**。外部数据集的歌曲标识符是否容易与 Spotify Track ID 对齐，是决定工程落地难度的最关键因素。

---

## 2. 备选开源数据集评估

### 2.1 ListenBrainz
**现代开源方案的首选，数据仍在持续更新。**

- **数据类型**：真实用户听歌事件（`user -> recording -> listens/timestamp`）。
- **优势**：
  - 目前仍在维护和更新，完全开放。
  - 数据规模庞大，官方明确支持下载和研究。
  - 基于 MusicBrainz，生态体系开放且结构化良好。
- **劣势**：
  - 使用 MusicBrainz ID，需要通过额外步骤映射到 Spotify ID。
  - 原始全量 Dump 极大，数据清洗和预处理的工程量较重。
- **适用场景**：希望做一个长期可持续、规模大的推荐系统基准。

### 2.2 Echo Nest Taste Profile Subset (Million Song Dataset)
**最适合快速搭建协同过滤 Baseline 的经典数据集。**

- **数据类型**：用户听歌频次（`user-song-playcount`）。
- **规模**：约 100 万用户，4800 万条交互记录，38 万首歌曲。
- **优势**：
  - 数据格式极其标准，天然适合直接跑 ALS（交替最小二乘法）、Item-CF 或 BPR 等协同过滤算法。
  - 学术界和工业界参考资料丰富。
- **劣势**：
  - 数据年代较老（2010年代初期）。
  - 歌曲 ID 是 Echo Nest 体系，需要做一次映射才能对齐到当前 Spotify 曲库。
- **适用场景**：想要以最快速度跑通“内容推荐 + 协同过滤”混合架构验证。

### 2.3 Last.fm / MHLD+
**适合大规模隐式反馈推荐。**

- **数据类型**：Last.fm 用户听歌历史（包含时间戳和频次）。
- **优势**：
  - 真实用户行为，规模庞大。
  - 提供丰富的长尾数据。
- **劣势**：
  - 版本众多且杂乱，不同版本的数据结构差异大。
  - 很多记录偏向 Artist（歌手）级别，精确到 Track（单曲）级别的映射有时不稳定。
  - 映射到 Spotify 曲库的成本偏高。
- **适用场景**：主要用于学术论文或算法实验。

### 2.4 Melon Playlist Dataset
**最接近 MPD（Spotify）形态的“歌单型”替代方案。**

- **数据类型**：歌单-歌曲共现（`playlist -> tracks + tags`）。
- **规模**：约 14.8 万个歌单，64.9 万首歌曲。
- **优势**：
  - 形态非常适合做 Playlist Co-occurrence（歌单共现）和 Item-CF。
- **劣势**：
  - 数据源是韩国音乐平台（Melon），音乐分布（K-Pop 偏多）可能与当前 Spotify 欧美主流曲库分布不一致。
  - ID 映射到 Spotify Track ID 会非常困难。
- **适用场景**：侧重研究歌单生成或标签推荐，但在本项目中落地难度大。

### 2.5 Spotify Million Playlist Dataset (MPD)
**最理想的选择，但官方已停止公开下载。**

- **数据类型**：歌单-歌曲共现（`playlist_id -> track_uri`）。
- **规模**：100 万个歌单，226 万首独特歌曲。
- **优势**：
  - `track_uri` 去掉前缀后就是当前库里的 `song_id`，**零映射成本**！
  - 完美契合项目“基于场景和氛围搭配”的需求。
- **现状**：
  - 官方挑战赛已结束，直接下载渠道关闭。如果能通过第三方镜像或科研渠道获取，则它是绝对的首选。

---

## 3. 落地建议与结论

根据当前项目的状态（已有完善的内容推荐链路，急需快速验证混合推荐架构）：

1. **如果有途径获取 MPD（即使是子集）**：**毫无疑问是第一选择**。它的零对齐成本能让你在一两天内直接写出高水准的 Item-CF 代码。
2. **如果完全依赖现存公开可下载数据，且想“快”**：推荐选择 **Taste Profile Subset**。它的三元组结构最适合快速训练一个混合模型（Hybrid Model），只需花一天时间解决 ID 映射问题。
3. **如果追求“最新、最开放”的数据生态**：推荐选择 **ListenBrainz**。需要前期投入更多精力做数据流管道（Pipeline）和 MusicBrainz-Spotify ID 对齐。

### 下一步行动指南
1. 选定一个数据集（如 Taste Profile 或 ListenBrainz）。
2. 下载一个小的 Sample（样本）。
3. 编写一个对齐脚本（如 `id_mapper.py`），测试其内部歌曲标识符能以多高的比例映射到 `data/processed/songs_processed.csv` 中。
4. 如果映射率超过 30%，则有足够的数据支撑协同过滤的落地。
