# 视界伴音推荐系统 - 下一步优化方案与分工（5人版）

> 本文档用于明日团队讨论，明确下一阶段的工作方向、任务拆解与人员分工（5人团队）。

---

## 一、项目现状回顾

### 1.1 已达成成果

| 维度                     | 现状                                                                                                                      |
| ------------------------ | ------------------------------------------------------------------------------------------------------------------------- |
| **数据集**         | 28,356 首去重歌曲，6 个流派（pop/rap/edm/r&b/rock/latin）                                                                 |
| **推荐链路**       | 双链路均打通：旧版（scene_profiles） + 新版（属性层 scene_attributes）                                                    |
| **核心优化已落地** | 流派软分数、加权特征相似度、软打分召回、推荐理由生成                                                                      |
| **视觉识别**       | CLIP ViT-B-32，91.67% 准确率（12 张测试图片）                                                                             |
| **评测体系**       | 场景重叠矩阵、质量摘要、特征中心距离、同场景漂移分析                                                                      |
| **服务化**         | Flask API：`/health`、`/scenes`、`/recommend/by_scene`、`/recommend/by_scene_attributes`、`/recommend/by_image` |

### 1.2 关键结论

- 新版属性层链路已具备足够稳定的工程形态，推荐质量优于旧版
- 协同过滤模块尚未开始实现，当前主要限制是外部交互数据集尚未稳定下载
- 推荐系统的主战场已经明确：**继续提高推荐质量、丰富评测维度，并为后续混合推荐做准备**

---

## 二、优化方案总览

共划分为 **6 个方向**，按优先级排列：

```
方向A: 推荐质量深度优化（P0）         ← 核心主战场
方向B: 视觉侧升级——CLIP 直接输出属性（P1）
方向C: 场景置信度分布输出（P2）
方向D: 协同过滤 / 混合推荐基础建设（P3）
方向E: 数据集扩充（P4）
方向F: 演示准备与课程展示（P5）
```

| A2   | **Top-5 / Top-10 人工 review** | 对 5 个场景 × 属性层链路做主观审查，标记「分数高但听感不符场景」的歌曲                                                                         | 问题歌曲清单 + 归因分析                  | ★★☆ | W1   |
| A3   | **属性映射权重微调**           | 基于 A1/A2 的结论，调整 `scene_attributes.json` 中的原型属性向量和 `attribute_mapper.py` 中的映射权重                                       | 优化后的配置文件                         | ★★☆ | W2   |
| A4   | **新增推荐质量评测指标**       | 增加：① 场景内歌曲特征方差 ② 相邻场景重叠合理性分析 ③ Top-10 歌手去重率                                                                      | 强化后的 `scene_overlap_evaluation.py` | ★★★ | W2   |
| A5   | **推荐理由质量评测**           | 审查 `generate_recommendation_reason()` 生成的可读解释是否准确反映歌曲与场景的匹配关系                                                        | 推荐理由优化                             | ★☆☆ | W3   |

### 👤 成员B：视觉算法工程师

**核心职责**：方向B（P1）CLIP属性化 + 方向C（P2）置信度分布

> **目标**：消除「场景标签分类错误 → 属性完全错误」的刚性传导，让图片直接映射到连续属性空间。

| 编号 | 任务                                   | 具体内容                                                                                                                                                | 产出                                                        | 工作量 | Week |
| ---- | -------------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------- | ------ | ---- |
| B1   | **实现 CLIP 属性打分器**         | 利用 `scene_attributes.json` 中已有的每属性 3 条正向 + 3 条反向 prompt，对 9 个属性逐一计算 `cos_sim(positive) - cos_sim(negative)`，归一化到 [0,1] | `scene_classifier.py` 新增 `classify_attributes()` 方法 | ★★★ | W1   |
| C1   | **修改 classify() 返回全部分布** | 将 `classify()` 从返回 `(best_scene, confidence)` 改为返回 `[(scene, score), ...]` 全量分布                                                       | `scene_classifier.py` 接口升级                            | ★☆☆ | W1   |
| B2   | **属性向量端到端集成**           | 将 CLIP 属性打分结果直接传入 `build_attribute_profile(attribute_scores=...)`，绕过场景标签                                                            | `recommend_by_image` 新增 `direct_attributes` 模式      | ★★☆ | W2   |
| C2   | **Top-2 混合召回**               | 取 Top-2 场景的属性向量做加权平均（以 CLIP 得分为权重），生成混合属性向量                                                                               | `recommender.py` 新增 `recommend_by_image_mixed()`      | ★★☆ | W2   |
| B3   | **属性层图片推荐测试**           | 用 12 张测试图片对比「标签中转」vs「直接属性」的 Top-10 结果差异                                                                                        | 对比评测报告                                                | ★☆☆ | W2   |
| C3   | **置信度可视化**                 | CLI / API 输出中展示场景得分分布                                                                                                                        | 输出格式扩展                                                | ★★☆ | W3   |
| B4   | **开放世界图片测试**             | 找 3-5 张不属于 5 个原型场景的图片（如超市、公园、地铁），验证属性向量的泛化能力                                                                        | 开放世界测试报告                                            | ★☆☆ | W3   |

**协作接口**：

- B1 完成后需要成员A的反馈（A1/A2）来优化属性映射
- C2 完成后需要与成员D协作实现 API 接口升级

## 七、关键决策点（明日讨论确认）

以下问题建议在会上确认：

1. **人员角色自愿选择** - 5个角色中是否有同学的偏好/专长匹配？
2. **协同过滤启动时机** - Week 3 的 D4/D5 是否提前到 Week 2？需要确认：

   - 是否有可用的公开数据集（Last.fm、Million Song Dataset）
   - 网络条件是否支持数据集下载
3. **新流派引入优先级** - E2 是否必须在 Week 2 完成？需要考虑：

   - 数据源是否已确定
   - 数据清洗/格式转换工作量
4. **前端复杂度** - F4 选择哪个版本？

   - 【极简版】纯 Flask 模板，无图表（★★★）
   - 【标准版】引入 Chart.js 做雷达图、置信度分布可视化（★★★）
5. **每周协作时间** - 建议的周一/周三/周五节奏是否可行？需要全员确认时间段。
6. **代码 Review 方式** - 选择哪种协作模式？

   - 选项A：每人独立分支，周五集中 PR Review
   - 选项B：主干开发，关键节点结对 Review
   - 选项C：核心模块多人共同开发（如 A+D 一起写混合排序）
